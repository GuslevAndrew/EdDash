# Публікація EdDash

Цей документ описує просту інфраструктуру для публікації EdDash у мережі.

## 1. Інфраструктура на пальцях

EdDash не повинен звертатися до ЄДЕБО при кожному відкритті сторінки. Правильна схема така:

```text
ЄДЕБО
  ↓
імпорт EdDash за розкладом або вручну
  ↓
наша база даних
  ↓
внутрішні запити EdDash
  ↓
сайт для користувачів
```

Тобто користувач відкриває сайт і бачить уже підготовлені дані з нашої бази. ЄДЕБО використовується тільки для оновлення цієї бази.

## 2. Що потрібно для запуску в мережі

Мінімальний набір:

1. Домен: `eddash.info`.
2. Хостинг для Next.js: Vercel, Render, Railway або інша платформа, яка підтримує Node.js.
3. Онлайн-база даних PostgreSQL: Supabase, Neon, Railway PostgreSQL або власний сервер PostgreSQL.
4. Змінні середовища на хостингу.
5. Механізм оновлення даних: спочатку ручний запуск імпорту, пізніше scheduled job.

## 3. Рекомендований перший варіант

Для першої публічної версії найпростіша схема:

```text
GitHub → Vercel → Next.js сайт
                 ↓
              PostgreSQL
```

Базу краще тримати окремо від сайту. Це дає нам:

- стабільне зберігання даних;
- можливість не втрачати базу після redeploy;
- нормальну роботу агрегованих запитів;
- простіше майбутнє масштабування.

SQLite залишаємо тільки для локальної розробки.

## 4. Чому не працювати напряму з ЄДЕБО

Прямі запити з сайту до ЄДЕБО при кожному відкритті сторінки не підходять, бо:

- зовнішній API може відповідати повільно;
- може тимчасово повертати HTML, файл або помилку;
- користувачі будуть чекати;
- ми не контролюємо навантаження;
- складно робити стабільні графіки й порівняння;
- можна випадково зробити надто багато запитів до джерела.

Тому EdDash має працювати з власною базою, а ЄДЕБО використовувати як джерело для імпорту.

## 5. Змінні середовища

Локально зараз використовується SQLite:

```env
DATABASE_URL="file:./dev.db"
EDBO_API_BASE_URL="https://registry.edbo.gov.ua"
EDBO_USER_AGENT="EdDash local analytics contact: contact@eddash.info"
```

Для продакшну потрібно буде замінити `DATABASE_URL` на PostgreSQL-рядок:

```env
DATABASE_URL="postgresql://USER:PASSWORD@HOST:PORT/DATABASE?schema=public"
EDBO_API_BASE_URL="https://registry.edbo.gov.ua"
EDBO_USER_AGENT="EdDash analytics contact: contact@eddash.info"
```

Реальні паролі й ключі не додаються в Git. Вони зберігаються тільки у змінних середовища хостингу.

## 6. Поточний стан проєкту

Зараз проєкт:

- запускається локально як Next.js-застосунок;
- використовує Prisma;
- локально працює з SQLite;
- має імпортери для відкритих даних ЄДЕБО;
- має дашборд, довідник закладів і довідник галузей та спеціальностей;
- ще не переведений на PostgreSQL.

## 7. Що треба зробити перед публікацією

Перед першим продакшн-деплоєм потрібно:

1. Створити PostgreSQL-базу.
2. Перемкнути Prisma datasource з SQLite на PostgreSQL.
3. Створити міграції для PostgreSQL.
4. Перевірити `npm run build`.
5. Завантажити базові довідники: регіони, спеціальності, заклади освіти.
6. Завантажити потрібні зрізи даних у PostgreSQL.
7. Вимкнути або захистити технічний розділ `/admin/import`.
8. Додати домен до хостингу.
9. Перевірити сайт на мобільному, планшеті й desktop.

## 8. Команди для перевірки перед деплоєм

```bash
npm install
npm run typecheck
npm run lint
npm run build
```

Для локальної бази:

```bash
npm run db:generate
npm run db:push:local
```

Для продакшн-бази після створення міграцій:

```bash
npm run db:migrate:deploy
```

## 9. Імпорт даних

Перший продакшн-імпорт краще робити контрольовано, не автоматично при відкритті сайту.

Орієнтовний порядок:

```bash
npm run edbo:regions
npm run edbo:specialities
npm run edbo:institutions
npm run edbo:import:years -- --years 2024,2025,2026
npm run edbo:outcomes:years -- --types entrants,graduates --years 2023,2024,2025
```

Фактичні параметри імпорту потрібно перевіряти перед запуском, щоб не робити зайві масові запити.

## 10. Адмін-розділ

Розділ `/admin/import` зараз технічний і локальний. Перед публікацією є три варіанти:

1. Тимчасово прибрати його з меню.
2. Закрити простим серверним паролем.
3. Винести імпорт у окремий protected workflow.

Для першого публічного запуску найкраще прибрати його з публічного меню і не давати випадковим користувачам запускати імпорт.

## 11. Наступний цільовий стан

```text
Користувачі
  ↓
eddash.info
  ↓
Vercel / Node.js hosting
  ↓
PostgreSQL / Supabase / Neon

Оновлення даних:
ЄДЕБО → scheduled import → PostgreSQL
```

Так ми отримуємо сайт, який швидко відкривається для користувачів і не залежить від стабільності ЄДЕБО в момент перегляду сторінки.

# EdDash: аудит MVP і план доробок

Дата: 16 червня 2026 року.

Цей документ фіксує, що в поточній MVP вже працює, що треба переробити перед публічним запуском, і які питання ще потрібно з'ясувати.

## Поточний стан

EdDash вже має робочу локальну основу:

- Next.js App Router застосунок українською мовою;
- локальну SQLite-базу через Prisma;
- імпорт демо-даних;
- тестовий клієнт API ЄДЕБО;
- імпортери регіонів, спеціальностей і здобувачів;
- внутрішні API для дашборду;
- фільтри, картки показників, графіки й таблицю з серверною пагінацією;
- довідкову сторінку `/specialities` з галузями знань і спеціальностями для вступників;
- технічну сторінку `/admin/import`.

Це нормальна MVP-модель для перевірки ідеї, але ще не production-сервіс.

## Архітектурне рішення: вища і фахова передвища освіта в одному дашборді

Фахову передвищу освіту додаємо не як окремий застосунок і не як окремий дашборд, а як рівень освіти у спільній аналітичній моделі EdDash.

Поточне рішення:

- `Institution` має поля `institutionTypeCode` і `institutionTypeName`;
- `1` означає вищу освіту;
- `9` означає фахову передвищу освіту;
- дашборд має фільтр "Рівень освіти": вища освіта, фахова передвища освіта або разом;
- таблиця показує рівень освіти для кожного запису;
- імпорт закладів бере дані з `/api/opendata/universities/?ut=1&exp=json` і `/api/opendata/universities/?ut=9&exp=json`;
- здобувачі ФПО імпортуються через поточний endpoint `/api/opendata/university-educators/` з `qf=9`.

Це перспективний шлях, бо дає змогу порівнювати вищу і фахову передвищу освіту в одному аналітичному просторі.

## Пріоритет 1: дані та методологія

Найважливіше зараз не дизайн, а коректність даних.

Що треба зробити:

- Чітко описати, що саме означає один запис `StudentSnapshot`.
- Вирішити, чи зберігаємо тільки загальну кількість здобувачів, чи також розрізи:
  - денна / заочна / вечірня;
  - бюджет / контракт;
  - спеціалізація;
  - форма власності;
  - підпорядкування.
- Додати сторінку або блок "Методологія", де пояснити джерело, дату зрізу, формулу підрахунку і відомі обмеження.
- Перевірити, чи код закладу з API ЄДЕБО є стабільним глобальним ідентифікатором.
- Перевірити, як ЄДЕБО поводиться з перейменуваннями закладів і спеціальностей.

Вже виправлено:

- Імпортер здобувачів агрегує рядки одного зрізу перед записом у базу, щоб кілька спеціалізацій не перезаписували одна одну.
- Додано локально згенерований довідник галузей і спеціальностей з офіційної таблиці постанови КМУ №266.

## Пріоритет 2: модель бази

Поточна схема достатня для MVP, але її треба посилити перед великим імпортом.

Проблемні місця:

- `Speciality.code` зараз унікальний, але в довіднику ЄДЕБО один код може мати різні історичні назви або ідентифікатори.
- `StudentSnapshot` не зберігає форми навчання і бюджет/контракт окремо.
- Немає таблиці для сирих або напівсирих імпортованих рядків, яка допомогла б аудитити помилки нормалізації.
- Немає явної ознаки `isDemo` для демо-даних.

Що варто додати:

- `ImportSourceRow` або схожу таблицю для збереження технічної інформації про імпорт.
- Поля для розрізів форм навчання, якщо вони важливі для аналітики.
- Окрему ознаку демо-даних або окремий тип імпорту, який легко фільтрувати.
- Нормальні міграції для PostgreSQL.

## Пріоритет 3: імпорт

Імпорт має стати надійним процесом, а не ручним скриптом.

Що треба доробити:

- Додати "план імпорту" для набору дат, рівнів і основ вступу без агресивних паралельних запитів.
- Додати dry-run режим.
- Додати детальніший звіт: скільки рядків отримано, скільки нормалізовано, скільки відкинуто і чому.
- Зробити кешування raw-відповідей для повторного аналізу без повторного звернення до ЄДЕБО.
- Додати захист від одночасного запуску двох імпортів.
- Додати лог помилок нормалізації по рядках.

## Пріоритет 4: дашборд і UX

Поточний дашборд функціональний, але ще потребує аналітичної глибини.

Що додати:

- Сторінку конкретного університету.
- Сторінку конкретної спеціальності.
- Порівняння 2-5 університетів.
- Порівняння регіонів.
- Фільтр "тільки реальні дані / тільки демо / усі".
- Пояснення, чому дані не знайдено, з підказками що змінити.
- Експорт таблиці в CSV.
- Збереження фільтрів у URL, щоб можна було ділитися посиланням.

Що переробити:

- Перевірити читабельність графіків на мобільних екранах.
- Скоротити дуже довгі назви в графіках, але показувати повну назву в tooltip.
- Додати skeleton/loading стани замість простого тексту.

## Пріоритет 5: адміністрування і безпека

Перед публікацією це обов'язково.

Що треба зробити:

- Закрити `/admin/import` авторизацією.
- Заборонити публічний POST на demo/test import.
- Винести імпорт у scheduled job або окремий backend-процес.
- Додати змінні середовища для режимів `local`, `staging`, `production`.
- Прибрати або ізолювати локальні Windows-специфічні залежності й `server.ps1`.

## Пріоритет 6: деплой

Найпростіший майбутній шлях:

```text
Vercel + Supabase Postgres + scheduled import
```

Більш контрольований шлях:

```text
VPS + PostgreSQL + Next.js process + cron
```

Перед деплоєм треба:

- перевести Prisma datasource з SQLite на PostgreSQL;
- створити production-міграції;
- перенести seed/import під PostgreSQL;
- додати auth для admin;
- налаштувати `DATABASE_URL` і секрети;
- перевірити build на чистому середовищі без локальних Windows-пакетів.

2. Додати таблицю відповідності спеціальностей 2015/2023.
3. Додати ознаку демо-даних або окремий спосіб відокремлення демо від реального імпорту.
4. Додати методологічний блок у застосунок.
5. Додати сторінку закладу освіти, яка працює і для ЗВО, і для ФПО.
6. Додати збереження фільтрів у URL.
7. Підготувати гілку для переходу на PostgreSQL.
